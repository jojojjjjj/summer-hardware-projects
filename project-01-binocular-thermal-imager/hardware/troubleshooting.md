# 硬件故障排除 | Hardware Troubleshooting

本文档帮助诊断和解决"ThermalEyes"双目手机热成像仪的硬件问题，按子系统分类，采用"症状→原因→解决方案"的格式。

> This document helps diagnose and resolve hardware issues with the "ThermalEyes" binocular thermal imager. Issues are organized by subsystem using a "Symptom → Cause → Solution" format.

## 一、电源问题 | Power Issues

### 问题 1.1：上电后无任何反应，LED 不亮

#### 症状 (Symptoms)
- 红色 LED 不亮
- 万用表测不到 3.3V
- MCU 不工作

#### 可能原因与解决方案 (Causes & Solutions)

**原因 1：USB 线是纯充电线，无数据传输功能**

> 部分廉价的 USB-C 线只有 VBUS 和 GND 两根线，没有数据线。这种线无法供电给 PCB（因为 Type-C 需要 CC 引脚来协商供电）。

解决方法：
1. 更换一根支持数据传输的 USB-C 线
2. 测试方法：用这根线连接手机和电脑，看电脑是否能识别手机

**原因 2：LDO 焊接方向错误或型号错误**

> AMS1117-3.3 的 SOT-223 封装有明确的引脚定义。焊反了不会有输出。

解决方法：
1. 查看 AMS1117 数据手册，确认引脚定义：
   - 左脚：GND（或 ADJ）
   - 中间大焊盘 + 右脚：VOUT（输出）
   - 标记面朝上时，从左到右依次是 GND、VOUT、VIN
2. 实际上 AMS1117 丝印面从左到右是 GND、VOUT、VIN
3. 测量输入端 (VIN)：应有约 5V（来自 USB VBUS）
4. 如果输入正常但无输出，更换 LDO 芯片

**原因 3：3.3V 与 GND 短路**

> 短路会导致 LDO 进入保护模式，无电压输出。AMS1117 有过流保护和过热保护。

解决方法：
1. 拔掉 USB 电源
2. 万用表调到电阻档（或蜂鸣档）
3. 测量 3.3V 网络和 GND 之间的电阻
4. 正常值应为数千欧姆以上
5. 如果电阻接近 0 欧，说明有短路
6. 逐一检查 3.3V 网络上的元件：
   - 去耦电容是否短路（电容击穿）
   - MCU 引脚是否桥接
   - LED 是否接反

**原因 4：Type-C CC 下拉电阻未安装**

> USB Type-C 需要在 CC1/CC2 引脚各接 5.1K 下拉电阻到 GND，才能让对端（手机/充电器）识别并供电。没有这两个电阻，USB 不会提供 VBUS 电压。

解决方法：
1. 确认 CC1 和 CC2 各有一个 5.1K 电阻接到 GND
2. 安装后重新上电测试

### 问题 1.2：3.3V 电压偏低（低于 3.0V）

#### 症状
- MCU 运行不稳定
- I2C 通信间歇性失败
- USB 连接频繁断开

#### 可能原因与解决方案

**原因 1：LDO 过载**

> AMS1117-3.3 最大输出 800mA。如果总电流超过这个值，输出电压会下降。

解决方法：
1. 断开 MLX90640 和 OV2640，只保留 MCU
2. 测量 3.3V 是否恢复正常
3. 逐个恢复连接，找出过载元件
4. 如果确实电流过大，考虑使用更大电流的 LDO（如 RT9018）

**原因 2：LDO 输出电容缺失或损坏**

> LDO 需要输出电容来保持稳定性。AMS1117 至少需要 10uF 的输出电容。

解决方法：
1. 确认 LDO 输出端有 10uF 电容
2. 用万用表测量电容是否短路或开路
3. 更换电容

**原因 3：USB 供电电压低**

> 某些 USB 口（特别是老旧电脑的前置 USB 口）可能供电电压不足 5V。

解决方法：
1. 测量 USB VBUS 电压：正常应为 5.0V ± 0.25V
2. 如果电压低，尝试其他 USB 口
3. 使用带外部电源的 USB Hub

---

### 问题 1.3：LDO 发热严重

#### 症状
- AMS1117 很烫手（超过 60°C）
- 工作一段时间后系统不稳定

#### 可能原因与解决方案

**原因：压降过大导致功耗高**

> AMS1117 压降约 1.7V (5V→3.3V)，电流 200mA 时功耗约 0.34W。如果电流更大，功耗更高。

解决方法：
1. 正常温热是正常的（40-50°C），不必担心
2. 如果烫手（超过 60°C），检查是否有短路
3. 在 LDO 底部的 PCB 上增加散热焊盘
4. 考虑换用效率更高的开关稳压器（如 MP2143），但成本更高

## 二、MCU 最小系统问题 | MCU Minimum System Issues

### 问题 2.1：ST-Link 无法连接 MCU

#### 症状
- STM32CubeIDE 提示"Error: ST-Link not connected"或"No device found"
- 无法读取芯片 ID
- 无法烧录固件

#### 可能原因与解决方案

**原因 1：SWD 接线错误**

> SWD 只需要 4 根线：SWDIO (PA13)、SWCLK (PA14)、GND 和可选的 3.3V（参考电压）。

解决方法：
1. 确认接线：
   - ST-Link SWDIO → MCU PA13 (Pin 37)
   - ST-Link SWCLK → MCU PA14 (Pin 38)
   - ST-Link GND → PCB GND
2. 不要连接 ST-Link 的 3.3V 到 PCB（避免与 USB 供电冲突）
3. 检查排线是否松动

**原因 2：MCU 焊接方向错误**

> LQFP48 封装的第 1 脚标记是一个小圆点或缺口。如果焊反了，所有引脚都是错的。

解决方法：
1. 检查芯片上的第 1 脚标记（圆点）
2. 对比 PCB 丝印上的 Pin 1 标记
3. 如果焊反了，需要用热风枪拆下重焊

**原因 3：NRST 引脚电平异常**

> NRST (Pin 7) 应通过 10K 电阻上拉到 3.3V。如果 NRST 被拉低，MCU 持续处于复位状态。

解决方法：
1. 测量 NRST 引脚电压：应为 3.3V
2. 如果为 0V：检查上拉电阻是否焊接正确
3. 检查复位按键是否卡住（短路到 GND）

**原因 4：BOOT0 引脚电平异常**

> BOOT0 (Pin 28) 应通过 10K 电阻下拉到 GND。如果 BOOT0 为高电平，MCU 会进入系统存储器启动模式而不是用户 Flash。

解决方法：
1. 测量 BOOT0 引脚电压：应为 0V（或接近 0V）
2. 如果为 3.3V：检查下拉电阻是否焊接
3. 注意：如果要进入 Bootloader 模式烧录（不使用 ST-Link），可以临时将 BOOT0 拉高

**原因 5：MCU 引脚桥接（连锡）**

> LQFP48 引脚间距仅 0.5mm，焊接时容易产生桥接。

解决方法：
1. 用放大镜或台式显微镜仔细检查每个引脚
2. 重点关注 VDD/VSS 引脚附近的桥接
3. 如发现桥接：涂助焊剂，用烙铁重新拖焊
4. 顽固桥接：用吸锡带吸走多余焊锡

### 问题 2.2：固件烧录后 MCU 不运行

#### 症状
- 烧录成功但程序不运行
- LED 不闪烁
- USB 不工作

#### 可能原因与解决方案

**原因 1：晶振未起振**

> STM32F411 需要外部 8MHz 晶振作为 HSE，通过 PLL 倍频到 100MHz。如果晶振不起振，系统时钟配置会失败。

解决方法：
1. 用示波器测量 PH0 (Pin 5) 和 PH1 (Pin 6)：应看到 8MHz 正弦波
2. 如果没有波形：
   - 检查晶振是否焊接正确
   - 检查 20pF 负载电容是否安装
   - 更换晶振试试
3. 临时方案：修改固件使用内部 HSI (16MHz) 时钟

**原因 2：系统时钟配置错误**

> 如果 PLL 配置超出芯片规格（F411 最高 100MHz），MCU 会卡在时钟初始化。

解决方法：
1. 检查 STM32CubeIDE 中的时钟树配置
2. 确认 SYSCLK 不超过 100MHz
3. 确认 USB 时钟为 48MHz（USB FS 需要精确的 48MHz 时钟）

**原因 3：固件使用了错误的芯片型号**

> STM32F401 和 STM32F411 的 Flash 大小和外设配置略有不同。

解决方法：
1. 确认 CubeIDE 工程中选择的是 STM32F411CEUx
2. 检查链接脚本中的 Flash 大小设置

## 三、I2C / MLX90640 问题 | I2C / MLX90640 Issues

### 问题 3.1：I2C 总线扫描不到 MLX90640

#### 症状
- I2C 扫描程序返回地址列表中没有 0x33
- 或扫描程序挂起（无限等待）

#### 可能原因与解决方案

**原因 1：SCL / SDA 接反**

> I2C 的 SCL 和 SDA 接反是新手最常犯的错误。MLX90640 的 SCL 和 SDA 引脚必须正确对应。

解决方法：
1. 确认接线：
   - STM32 PB6 (I2C1_SCL) → MLX90640 SCL
   - STM32 PB7 (I2C1_SDA) → MLX90640 SDA
2. 用万用表的二极管档验证导通

**原因 2：上拉电阻缺失**

> I2C 是开漏输出，需要外部上拉电阻。没有上拉电阻，总线永远为低电平，通信无法进行。

解决方法：
1. 断电状态下，测量 SCL 和 SDA 到 3.3V 的电阻：应为 4.7K
2. 如果为无穷大（开路）：缺少上拉电阻，需要添加
3. 注意部分 MLX90640 模块已自带 10K 上拉，但可能不够强（推荐 4.7K）

**原因 3：MLX90640 供电异常**

> MLX90640 的工作电压是 3.3V (2.7V-3.6V)。如果供电电压不对，芯片不会响应。

解决方法：
1. 测量 MLX90640 的 VDD 引脚电压：应为 3.3V
2. 如果为 0V：检查供电线路
3. 如果为 5V：立即断电！MLX90640 不支持 5V，可能已损坏
4. 如果电压偏低（低于 2.7V）：检查 LDO 输出

**原因 4：MLX90640 I2C 地址不是 0x33**

> MLX90640 的默认 I2C 地址是 0x33（7位地址），但某些模块可能使用了不同的地址。

解决方法：
1. 进行全地址扫描 (0x03 到 0x77)
2. 如果发现设备在其他地址，修改固件中的地址定义
3. MLX90640 的地址可以通过硬件配置修改

**原因 5：I2C 速度过快**

> MLX90640 支持 I2C 快速模式 (400kHz)，但某些情况下信号质量不好时需要降速。

解决方法：
1. 将 I2C 速度从 400kHz 降到 100kHz
2. 如果 100kHz 能工作但 400kHz 不行，检查：
   - 走线长度和布线质量
   - 上拉电阻阻值（4.7K 适用于 400kHz，如果走线长可以用 2.2K）
   - 是否有总线电容过大（线太长或挂了太多设备）

### 问题 3.2：MLX90640 能扫描到但读取数据全为 0 或异常

#### 症状
- I2C 地址扫描能找到 0x33
- 读取的像素数据全为 0 或全为同一个值
- 计算出的温度值不合理（负数或极高值）

#### 可能原因与解决方案

**原因 1：读取了错误的寄存器地址**

> MLX90640 的帧数据存储在 0x0400-0x07FF 的寄存器区域。如果读取了错误的地址，数据无意义。

解决方法：
1. 确认固件中读取的寄存器起始地址为 0x0400
2. 每帧需要读取 832 个字节（32x24 像素 x 2 字节/像素）
3. 参考 MLX90640 数据手册的帧结构

**原因 2：MLX90640 未正确初始化**

> MLX90640 需要先写入配置寄存器才能正常工作。

解决方法：
1. 检查初始化序列：
   - 写入 0x800D: 0x0080 (刷新速率设置)
   - 写入 0x800D: 0x0010 (设置 2fps) 或其他帧率
2. 确认初始化函数被正确调用

**原因 3：帧未就绪就读取**

> MLX90640 的数据更新是异步的。如果在帧更新过程中读取，会得到混合数据。

解决方法：
1. 读取状态寄存器 (0x8000) 检查新帧是否就绪
2. 在帧就绪后再读取像素数据

## 四、摄像头问题 | Camera Issues

### 问题 4.1：OV2640 无图像输出

#### 症状
- APP 上显示全黑画面
- DCMI 接口收不到数据
- SCCB 无法读取芯片 ID

#### 可能原因与解决方案

**原因 1：FPC 排线插入方向错误**

> OV2640 的 FPC 排线有正反面之分。金手指面（有金属触点的一面）应朝向 FPC 座子的触点。

解决方法：
1. 打开 FPC 座子翻盖
2. 取出排线
3. 确认金手指面朝向座子内部的金属触点
4. 重新插入并扣紧翻盖
5. 轻拉排线确认已锁定

**原因 2：FPC 排线未插紧**

> 排线如果没有完全推入，只有部分触点连接。

解决方法：
1. 打开翻盖
2. 将排线推到最深处
3. 压紧翻盖
4. 用万用表检查排线两端导通性

**原因 3：OV2640 PWDN 引脚为高电平**

> OV2640 的 PWDN 引脚（省电控制）为高电平时，摄像头进入断电模式，不会响应任何通信。

解决方法：
1. 确认 PWDN 引脚连接到 MCU 的 GPIO 或直接拉低到 GND
2. 如果连接到 GPIO：在固件中将该 GPIO 设为低电平输出
3. 如果直接拉低：确认电阻/焊点连接正确

**原因 4：SCCB 通信失败**

> SCCB (Serial Camera Control Bus) 类似 I2C，但时序略有不同。

解决方法：
1. 确认 PA8 (SCL) 和 PA9 (SDA) 有 4.7K 上拉电阻
2. 尝试读取芯片 ID 寄存器 (地址 0x0A/0x0B)：应返回 0x2642
3. 如果读不到：
   - 降低 SCCB 时钟频率
   - 检查接线
   - 尝试使用软件 I2C 替代硬件 I2C

**原因 5：DCMI 时序配置不匹配**

> DCMI 接口的时序必须与 OV2640 的输出时序匹配。

解决方法：
1. 确认 OV2640 配置为 UVC 输出模式（非 JPEG 模式）
2. 检查 DCMI 的像素时钟极性 (PCLK polarity)
3. 检查 VSYNC 和 HSYNC 的极性设置
4. OV2640 的默认极性：VSYNC 低有效，HREF 高有效，PCLK 上升沿有效

### 问题 4.2：OV2640 图像花屏或有条纹

#### 症状
- 画面有规律性条纹
- 颜色不正常
- 画面撕裂

#### 可能原因与解决方案

**原因 1：DCMI 数据线接错**

> 8 位并行数据线 D0-D7 的顺序必须正确。接错会导致像素值混乱。

解决方法：
1. 逐一检查 D0-D7 的接线
2. 确认接线与 wiring-guide.md 中的映射表一致

**原因 2：OV2640 输出格式配置错误**

> OV2640 支持多种输出格式（RGB565、YUV422、JPEG 等）。如果格式不匹配，图像会花屏。

解决方法：
1. 确认 OV2640 配置为 RGB565 或 YUV422 输出
2. MCU 端的 DCMI 配置必须与传感器输出格式匹配

**原因 3：帧率不匹配导致画面撕裂**

> 如果 MCU 读取帧的速度与 OV2640 输出帧的速度不同步，会出现画面撕裂。

解决方法：
1. 使用 DCMI 的硬件同步信号（VSYNC/HREF）
2. 确认使用 DMA 接收帧数据
3. 使用双缓冲避免读写冲突

## 五、USB 通信问题 | USB Communication Issues

### 问题 5.1：手机不识别 USB 设备

#### 症状
- USB 线连接后手机无反应
- 手机不弹出 USB 设备通知
- APP 无法找到 USB 设备

#### 可能原因与解决方案

**原因 1：D+ 上拉电阻未安装或阻值错误**

> USB 全速设备需要在 D+ 线上接 1.5K 上拉电阻到 3.3V。这个电阻告诉主机（手机）有设备连接。没有这个电阻，手机完全不会感知到设备。

解决方法：
1. 确认 PA12 (USB_DP) 与 3.3V 之间有 1.5K 电阻
2. 测量 D+ 线的空闲电压：应约为 3.3V
3. 确认不是接到 5V（会损坏 MCU 的 USB 引脚）

**原因 2：USB Type-C CC 下拉电阻缺失**

> Type-C 接口需要在 CC1 和 CC2 引脚各接 5.1K 下拉电阻到 GND。没有这些电阻，手机不会提供 VBUS 电压。

解决方法：
1. 确认 CC1 和 CC2 各有 5.1K 下拉电阻到 GND
2. 测量 CC 引脚电压：应约为 0V（通过电阻拉低）

**原因 3：使用了纯充电线**

> 很多 USB-C 线只有电源线，没有 D+/D- 数据线。使用这种线无法通信。

解决方法：
1. 更换支持数据传输的 USB-C 线
2. 测试方法：用这根线连接手机和电脑，确认电脑能识别手机

**原因 4：USB D+/D- 接反**

> D+ (PA12) 和 D- (PA11) 接反会导致 USB 无法枚举。

解决方法：
1. 确认 PA12 → USB D+ (通过 22R 电阻)
2. 确认 PA11 → USB D- (通过 22R 电阻)
3. 用万用表验证导通

**原因 5：固件中 USB 外设未正确初始化**

> USB 需要正确配置时钟（48MHz）和中断。

解决方法：
1. 确认 USB 时钟为精确的 48MHz（由 PLL 产生）
2. 检查 USB 中断是否使能
3. 尝试连接电脑，查看设备管理器中是否出现未知设备
4. 如果电脑上能看到设备但手机上不行，可能是手机的 USB Host 驱动问题

### 问题 5.2：USB 连接不稳定，频繁断开

#### 症状
- 手机偶尔识别到设备但很快断开
- 数据传输中突然中断
- 需要反复拔插才能连接

#### 可能原因与解决方案

**原因 1：USB 信号质量问题**

> USB 信号对走线质量敏感。D+/D- 走线不等长、阻抗不匹配都会导致通信不稳定。

解决方法：
1. 确认 D+ 和 D- 的 22R 串联电阻已安装
2. 检查 USB 走线是否有急转弯或过长的走线
3. 尝试更短的 USB 线

**原因 2：供电不足导致电压波动**

> USB 供电电流不足时，3.3V 可能波动，导致 MCU 复位或 USB 断开。

解决方法：
1. 用示波器测量 3.3V 是否有波动（应稳定在 3.3V ± 0.05V）
2. 增加更大的输出电容（从 10uF 增加到 22uF 或 47uF）
3. 确保 LDO 输入端也有足够的电容

**原因 3：USB 中断优先级冲突**

> 如果 USB 中断被其他更高优先级的中断阻塞，会导致 USB 超时断开。

解决方法：
1. 将 USB 中断优先级设为较高（数值较小）
2. 确保中断处理函数尽可能短

## 六、系统集成问题 | System Integration Issues

### 问题 6.1：MLX90640 和 OV2640 数据不同步

#### 症状
- 热成像图像和可见光图像不重叠
- 融合图像有明显的位置偏差

#### 可能原因与解决方案

**原因：两个传感器帧率不同步**

> MLX90640 最高 32fps（或更低），OV2640 可达 30fps 或更高。帧率不同导致时间上的不一致。

解决方法：
1. 在固件中使用时间戳标记每帧数据
2. 在 APP 端做帧同步（匹配最近时间戳的帧）
3. 调整 MLX90640 的帧率与 OV2640 匹配

### 问题 6.2：APP 显示延迟大

#### 症状
- 手势或动作后，画面延迟 1-2 秒才更新
- 体验不流畅

#### 可能原因与解决方案

**原因 1：USB 传输带宽不足**

> USB FS (Full-Speed) 的理论带宽为 12Mbps，实际有效带宽约 8-9Mbps。

解决方法：
1. 优化数据打包格式，减少每帧的传输数据量
2. MLX90640 的 32x24 数据很小（768 像素 x 2 字节 = 1536 字节），不是瓶颈
3. OV2640 的图像较大，可能需要降低分辨率或使用 JPEG 压缩

**原因 2：APP 端图像处理耗时**

> OpenCV 的图像融合算法如果未优化，可能耗时较长。

解决方法：
1. 使用 NDK + C++ 实现融合算法，比 Java 快
2. 减少图像处理步骤
3. 降低处理分辨率

### 问题 6.3：温度测量不准确

#### 症状
- 测量温度与实际温度偏差大
- 不同位置的温度差异不合理

#### 可能原因与解决方案

**原因 1：MLX90640 未做发射率校准**

> MLX90640 默认发射率为 1.0，适用于大多数非金属表面。测量反光表面（如金属、玻璃）时会有偏差。

解决方法：
1. 在 APP 中添加发射率调节功能
2. 测量人体皮肤：发射率约 0.98
3. 测量金属表面：发射率约 0.1-0.3，读数会严重偏低

**原因 2：传感器自发热**

> MLX90640 长时间工作后自身会发热，影响测量精度。

解决方法：
1. MLX90640 内置了温度补偿，读取芯片温度做补偿
2. 确保传感器周围有足够的散热空间
3. 刚上电时等 1-2 分钟让传感器温度稳定

**原因 3：镜头脏污**

> MLX90640 的硅镜头表面如果有指纹或灰尘，会影响红外透过率。

解决方法：
1. 用气吹清理镜头
2. 不要用手触摸镜头
3. 不要用酒精或溶剂擦拭

## 九、常见问题 FAQ | Frequently Asked Questions

### Q1: 可以用 STM32F103 代替 STM32F411 吗？

可以，但有以下限制：
- STM32F103 的 USB 是全速 (FS)，可以工作但传输速度比 F411 的 USB 略慢
- STM32F103 没有 DCMI 接口，需要用 GPIO 模拟并行摄像头接口，代码更复杂
- Flash 只有 64KB，空间比较紧张
- 建议还是使用 STM32F411，教程和固件都基于这个芯片

### Q2: MLX90640 可以换成 MLX90614 吗？

不建议。MLX90614 是单点红外温度传感器（1 个像素），不是热成像阵列。本项目需要 32x24 像素的热成像阵列才能生成图像。

### Q3: 可以同时用两个 MLX90640 吗？

可以。MLX90640 支持修改 I2C 地址。将两个 MLX90640 设为不同地址（0x33 和 0x32），连接在同一条 I2C 总线上即可。但固件需要做相应修改。

### Q4: OV2640 和 OV5640 的区别是什么？

| 特性 | OV2640 | OV5640 |
|------|--------|--------|
| 分辨率 | 200万像素 (1600x1200) | 500万像素 (2592x1944) |
| 接口 | DVP 8位并行 | DVP 8位并行 / MIPI CSI |
| 价格 | ¥8-15 | ¥15-25 |
| 自动对焦 | 不支持 | 部分型号支持 |
| 驱动复杂度 | 简单 | 稍复杂 |

本项目推荐 OV2640，成本低、驱动简单。如果需要更高画质可以升级到 OV5640，但需要修改固件中的 SCCB 配置。

### Q5: 为什么不能用 iOS 手机？

iOS 设备的 USB Host 支持受限。Apple 的 Lightning/USB-C 接口对外部 USB 设备的支持需要 MFi 认证。Android 设备原生支持 USB Host 模式，可以直接连接 USB 外设。如果需要 iOS 支持，可以考虑使用 BLE (蓝牙低功耗) 传输数据，但需要大幅修改硬件和软件设计。

### Q6: PCB 可以不用沉金吗？

可以。无铅喷锡 (HASL) 也可以用，而且更便宜。沉金 (ENIG) 的优点是焊盘更平整，对细间距贴片元件（如 LQFP48）的焊接更友好。如果你对自己的焊接技术有信心，用 HASL 也没问题。

### Q7: 烙铁温度应该设置多少？

- **贴片焊接（0402/0805 电阻电容）**：320-340°C
- **LQFP48 芯片拖焊**：340-360°C
- **插件元件（排针、按键）**：350-370°C
- **拆焊**：380-400°C（短时间使用）

温度过低会导致焊锡不熔化或冷焊（虚焊），温度过高会损坏元件或焊盘。新手建议从 340°C 开始。

### Q8: 焊完之后需要清洗吗？

建议清洗。助焊剂残留物长期会腐蚀焊点。用无水乙醇（纯度 95% 以上）和棉签擦拭焊点区域。清洗后自然晾干或用压缩空气吹干。
