# 详细评分标准 | Detailed Grading Rubric

## 评分维度 | Grading Dimensions

### 1. 技术实现 (35%) | Technical Implementation

| 等级 Grade | 分数 Score | 标准 Criteria |
|-----------|-----------|--------------|
| A - 优秀 | 90-100 | 所有核心功能正常（时间/NTP、天气API、OLED显示），代码质量高，有拓展功能（PC监控或B站粉丝数），无明显 bug |
| B - 良好 | 75-89 | 核心功能完整，偶有小 bug 但已修复，代码结构清晰 |
| C - 合格 | 60-74 | 基本功能可用（至少时间+天气能显示），存在已知 bug 但已记录在文档中 |
| D - 待改进 | < 60 | 核心功能未完成或无法运行 |

### 2. 文档质量 (9%) | Documentation

| 等级 | 分数 | 标准 |
|------|------|------|
| A | 90-100 | README 完整，代码有清晰注释，接线图/原理图规范，有调试日志和问题记录 |
| B | 75-89 | README 完整，代码有一定注释，接线图基本清晰 |
| C | 60-74 | README 存在但内容不完整，代码注释较少 |
| D | < 60 | 几乎没有文档 |

### 3. 演示展示 (17%) | Presentation

| 等级 | 分数 | 标准 |
|------|------|------|
| A | 90-100 | 演示流畅，实物正常运转，技术讲解深入，Q&A 对答如流，时间控制精准 |
| B | 75-89 | 演示基本顺畅，实物能正常工作，技术讲解清晰，能回答大部分问题 |
| C | 60-74 | 演示有卡顿，实物部分功能异常，讲解不够清晰 |
| D | < 60 | 无法完成演示或实物完全不能工作 |

### 4. 进度汇报 (17%) | Check-ins

| 等级 | 分数 | 标准 |
|------|------|------|
| A | 90-100 | 两次汇报均按时提交，内容详实，问题分析深入，反思有深度，计划清晰可行 |
| B | 75-89 | 两次汇报按时提交，内容完整，有基本的分析和反思 |
| C | 60-74 | 仅提交一次或两次都较敷衍 |
| D | < 60 | 未提交 |

### 5. 团队协作 (9%) | Collaboration

| 等级 | 分数 | 标准 |
|------|------|------|
| A | 90-100 | Git 提交记录规律，分工明确，互帮互助，积极沟通 |
| B | 75-89 | Git 有提交记录，分工基本合理 |
| C | 60-74 | Git 提交较少或不规律 |
| D | < 60 | 无 Git 记录，分工混乱 |

### 6. Demo 网站 (13%) | Project Website

| 等级 | 分数 | 标准 |
|------|------|------|
| A | 90-100 | 网站内容完整（介绍、功能展示、技术架构、心得），视觉设计美观，已部署可访问 |
| B | 75-89 | 内容基本完整，已部署可访问 |
| C | 60-74 | 内容不完整或未成功部署 |
| D | < 60 | 无 Demo 网站 |

---

## 总分计算 | Total Score Calculation

```
总分 = 技术实现×0.35 + 文档质量×0.09 + 演示展示×0.17 + 进度汇报×0.17 + 团队协作×0.09 + Demo网站×0.13
```

> 注：独立完成项目的学员，"团队协作" 9% 转换为 "自主学习能力" 评分，考察独立解决问题和自主查阅资料的能力。
