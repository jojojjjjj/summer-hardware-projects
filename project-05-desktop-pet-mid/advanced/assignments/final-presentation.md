# 最终展示要求（进阶方向）| Final Presentation Requirements (Advanced Track)

## 概述 | Overview

在课程最后一天，每位学员（或每组）将有 **10 分钟时间**（5 分钟演示 + 5 分钟 Q&A），向全班展示你亲手打造的 AI 桌面宠物。

On the final day, each student (or team) will have **10 minutes** (5 min presentation + 5 min Q&A) to present their hand-built AI desktop pet to the class.

---

## 演示结构 | Presentation Structure

### Part 1: 项目介绍（1 分钟）| Project Introduction

**必须涵盖的内容：**
- 你的桌面宠物叫什么名字？有什么性格特点？
- 一句话介绍项目：基于 ESP32-S3 + Python 后端的 AI 桌面宠物
- 为什么选择进阶方向？它和基础方向有什么不同？
- 你的桌面宠物的核心亮点是什么？

**可选加分：**
- 展示你的桌面宠物的个性化设计（自定义外观、独特对话风格）

---

### Part 2: 硬件与组装展示（1 分钟）| Hardware Showcase

**必须展示的内容：**

1. **组装完成的桌面宠物实物**
   - 指出各模块位置：ESP32-S3、屏幕、摄像头、舵机、麦克风、喇叭
   - 展示整体外观和接线

2. **组装过程回顾** -- 用 2-3 张照片快速回顾：
   - 焊接过程
   - 舵机安装
   - 首次上电点亮

---

### Part 3: 核心功能演示（3 分钟）| Live Feature Demo

**必须演示的功能（4 项）：**

#### 1. 语音交互与 AI 对话（重点展示，约 90 秒）

这是进阶方向的核心亮点。必须展示完整的语音对话流程：

1. 对着桌面宠物说一句话
2. 宠物听到后展示"思考"表情
3. AI 大模型生成回复
4. 宠物通过喇叭语音回复
5. 宠物展示与回复内容匹配的表情

**展示要点：**
- 语音识别准确性（能正确转写你的话）
- AI 回复的自然度（不是机械的模板回复）
- 语音合成的清晰度（能清楚听懂回复内容）
- 端到端延迟（从说话到听到回复的时间）

#### 2. 面部表情系统（约 30 秒）

展示桌面宠物的表情切换：
- 开心（微笑）
- 悲伤（失落）
- 惊讶
- 思考
- 困倦/睡眠

展示表情与动作的联动效果（如开心时点头、悲伤时低头）。

#### 3. 运动/舵机动作（约 30 秒）

展示桌面宠物的肢体运动：
- 点头（表示同意）
- 摇头（表示否认）
- 挥手（打招呼）
- 其他自定义动作

#### 4. Web 控制面板（约 30 秒）

通过浏览器展示 Web 控制功能：
- 打开控制面板页面
- 远程触发表情切换
- 远程控制舵机运动
- 查看摄像头画面（如集成）
- 触发 AI 对话

---

### Part 4: 技术总结（2 分钟）| Technical Summary

选择以下 **一个** 话题深入讲解：

**话题 A：多模态 AI 管线架构**

画出完整的多模态 AI 管线数据流图，包括：

```
用户语音 → [麦克风 I2S采集] → ESP32 → [HTTP上传] → Python后端
    → [ASR语音识别] → 文本 → [LLM大模型] → 回复文本
    → [TTS语音合成] → 音频 → [HTTP下载] → ESP32 → [I2S播放] → 喇叭
                                                    ↓
                                              [表情动画] → 屏幕
                                              [舵机动作] → 四肢
```

解释各环节的技术选型和实现细节：
- 为什么选择 Python 后端而不是 ESP32 直接调 API？
- ASR 使用什么服务？（通义听悟 / DashScope）
- LLM 使用什么模型？（通义千问 / 其他）
- TTS 使用什么服务？
- 如何处理端到端延迟？

**话题 B：Arduino 固件架构设计**

展示你的 Arduino 项目结构，并解释每个模块的职责：

```
firmware/
├── desktop_pet.ino        # 主入口
├── screen_controller.cpp  # 屏幕与表情控制
├── servo_controller.cpp   # 舵机与动作控制
├── camera_handler.cpp     # 摄像头功能
├── audio_handler.cpp      # 麦克风与喇叭
├── wifi_manager.cpp       # Wi-Fi 与后端通信
└── pet_personality.cpp    # 宠物性格与行为逻辑
```

解释：
- 各模块如何通信？（全局变量、回调函数、队列）
- FreeRTOS 任务如何分配？（哪些功能用 task，哪些在 loop 中）
- 内存管理策略？（PSRAM 使用、缓冲区大小）

**话题 C：一个技术难点的攻克过程**

选择你在开发过程中遇到的一个具体问题，详细展开：
- Python 后端与 ESP32 的通信调试
- 音频数据的有损/无损传输
- 多模态 AI 管线的延迟优化
- TFT_eSPI 屏幕驱动的配置问题
- 舵机抖动/供电不足的解决方案

## Q&A 准备 | Q&A Preparation

以下是可能被问到的问题，提前准备好答案：

| 可能的问题 | 准备方向 |
|------------|----------|
| 为什么进阶方向使用 Python 后端，而不是 ESP32 直接调 API？ | ESP32 内存有限，ASR/TTS 处理复杂；Python 后端可扩展性强，方便切换不同 AI 服务 |
| 你的多模态 AI 管线的延迟瓶颈在哪里？ | 通常是 ASR 或 LLM 推理环节；可通过流式响应、模型选择优化 |
| Arduino 框架和 ESP-IDF 有什么区别？ | Arduino 封装度高、上手快；ESP-IDF 更底层、功能更全但学习曲线陡 |
| 你如何管理 API Key 的安全性？ | 使用 .env 文件、不提交到 Git、后端中转避免 ESP32 直接暴露 |
| 如果让你继续开发，你最想加什么功能？ | 准备 1-2 个拓展方向（视觉识别、多设备联动、情感分析等） |
| 你的后端如何处理并发请求？ | FastAPI 基于 asyncio，天然支持异步并发 |
| ESP32 和后端之间的通信协议为什么选 HTTP/WebSocket？ | HTTP 用于简单请求/响应，WebSocket 用于实时双向通信（如音频流） |
| 为什么选择通义千问而不是其他大模型？ | DashScope API 支持多模态（ASR+LLM+TTS 集成）、国内访问稳定、有免费额度 |

### 备用故障预案 | Emergency Backup Plan

| 故障场景 | 预案 |
|----------|------|
| 桌面宠物无法开机 | 使用提前录制的功能演示视频 |
| Wi-Fi 无法连接 | 展示离线功能（表情、舵机），通过代码讲解在线功能原理 |
| Python 后端无法启动 | 提前录制后端 API 调用演示，展示 API 文档 |
| AI API 调用失败 | 展示之前录制的对话 Demo 视频，解释 API 调用原理 |
| 屏幕不显示 | 通过串口日志展示程序在后台运行，用录制视频替代 |
| 舵机不响应 | 展示舵机代码和控制逻辑，说明硬件故障原因 |

## 演示小贴士 | Presentation Tips

1. **提前准备演示话术** -- 不要即兴发挥，提前写好每个环节的过渡语
2. **展示真实效果** -- 如果某个功能有 bug，坦诚说明比假装完美更打动人
3. **控制时间** -- 提前练习 2 遍，确保 5 分钟演示不超时
4. **互动感** -- 邀请同学对桌面宠物说话，增加参与感
5. **备份一切** -- 录制所有功能的视频作为 Plan B
6. **技术深度** -- Q&A 时展示你对系统的深入理解，而不只是"能跑就行"
