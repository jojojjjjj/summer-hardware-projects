# UPenn ESE5160 课程设计思路抽象 | Abstracted Course Design Methodology

> 抽象自 UPenn ESE5160: IoT Edge Computing (Spring 2026)
> Adapted for: 高中生暑期硬件项目库 (Summer Hardware Project Repository)

---

## 一、课程设计总纲 | Overall Design Framework

ESE5160 是一门以**项目驱动（Project-based Learning）**为核心的课程，学生在 16 周内从零开始设计、制造并交付一个完整的 IoT 设备。其核心设计思路可以抽象为以下五个阶段：

```
[Phase 0: 概念与团队] → [Phase 1: 硬件设计] → [Phase 2: 制造发布] → [Phase 3: 固件开发] → [Phase 4: 集成交付]
```

每个阶段内部又遵循 **"理解 → 实践 → 验证 → 反思"** 的微循环。

## 二、渐进式脚手架模型 | Progressive Scaffolding Model

### 2.1 由浅入深的知识递进曲线

ESE5160 的作业序列呈现出清晰的**螺旋上升（Spiral Progression）**结构：

| 阶段 | 作业编号 | 核心任务 | 认知层级 |
|------|---------|---------|---------|
| Phase 0 | A00G | 选器件、画框图、组团队 | 识记/理解 |
| Phase 1 | A01G | 元器件验证、BOM、电源架构 | 理解/应用 |
| Phase 1 | A02G | 原理图设计 I | 应用 |
| Phase 1 | A03G | 原理图设计 II + BOMDOC | 应用/分析 |
| Phase 2 | A04G | PCB 布局 | 应用/分析 |
| Phase 2 | A05G | 制造发布 | 分析/评估 |
| Phase 3 | A06G | FreeRTOS & I2C | 应用/分析 |
| Phase 3 | A07G | 板卡调试 | 分析/评估 |
| Phase 3 | A08G | 云端通信 | 应用/分析 |
| Phase 3 | A09G | 驱动与应用逻辑 | 分析/评估 |
| Phase 4 | A10G | 审判日（中期评审） | 评估 |
| Phase 4 | A11G | 最终提交 | 创造 |
| Phase 4 | A12G | 口试答辩 | 评估/创造 |

**关键洞察**：每个作业的输出是下一个作业的输入，形成**严格的依赖链**。

### 2.2 双轨并行作业体系

ESE5160 巧妙地设计了**个人作业（P系列）**和**团队作业（G系列）**双轨：

- **个人作业 (Personal)**：聚焦工具技能训练（Altium教程、时钟与定时器、制造基础）
- **团队作业 (Group)**：聚焦系统级设计与集成

这种设计确保：
1. 个人能力不足不会拖垮团队进度
2. 工具技能在个人轨道提前练好，团队轨道专注系统
3. 每个学生都有可独立评估的成果

## 三、每份作业的标准化结构 | Standardized Assignment Template

每份 ESE5160 作业都遵循统一的结构模板：

```
# [作业编号] [作业标题]

## 课程信息 & 截止日期
## Learning Outcomes（学习成果）—— 明确、可衡量
## Errata（勘误）—— 持续更新的常见问题
## Required Equipment（所需设备/工具）

### 1. [核心任务 1]
### 2. [核心任务 2]
### ...

## Submission Requirements（提交要求）
## Rubric（评分标准）
```

**核心设计要点**：
1. **Learning Outcomes 在最前面**：学生先知道"学完这个我能做什么"
2. **Errata 机制**：承认作业不可能一次完美，预留持续修正空间
3. **每个任务都有具体的提交物**：不是"理解XX"，而是"提交XX文件/截图/代码"
4. **评分标准与学习成果一一对应**：可追溯、可量化

## 四、知识深度的三层设计 | Three-Layer Knowledge Depth

ESE5160 的每个作业实际上隐含了三个层次的知识要求：

### Layer 1: 知道是什么（Know-What）
- 这个器件/协议/工具是什么？
- 它的基本参数和特性是什么？
- 例：知道 I2C 是双线制半双工通信协议，有起始位、地址、ACK/NACK

### Layer 2: 知道为什么（Know-Why）
- 为什么要这样设计？
- 这个设计选择背后的权衡（Trade-off）是什么？
- 例：为什么 I2C 用开漏输出而不是推挽？因为多主机总线需要线与功能

### Layer 3: 知道怎么做（Know-How）
- 具体如何实现？
- 如何调试？如何排查问题？
- 例：如何用 HAL 库实现 I2C 读写，如何用示波器/逻辑分析仪抓包分析

**研究生课程要求三层全覆盖。高中生暑期项目要求调整为：**
- **Layer 1 必须**：每个方向的基础概念
- **Layer 2 重点**：设计选择的"为什么"——这正是能和老师进行技术对话的关键
- **Layer 3 适度**：能完成基本实现，不需要精通

## 五、评审与反馈机制 | Review & Feedback Mechanism

ESE5160 的评审体系设计值得借鉴：

### 5.1 多轮评审制度
- **A04G → Layout Review**：课上 PCB Layout 评审
- **A05G → Release Review**：制造前终审
- **A07G → Board Bringup**：硬件调试评审
- **A10G → Judgement Day**：中期总评审
- **A12G → Exit Interview**：口试答辩

### 5.2 评审的三个维度
1. **功能正确性**：做出来的东西能不能用？
2. **设计合理性**：为什么选择这个方案而不是那个？
3. **可维护性**：别人能否看懂你的设计/代码？

### 5.3 同行评估
- 期中团队反馈（Mid-semester Team Feedback）
- 期末队友评价（End of Semester Teammate Feedback）
- 确保团队协作中没有人"搭便车"

## 六、适配高中生的调整策略 | Adaptation for High School Students

### 6.1 降低维度，保留结构

| UPenn 原版 | 高中生适配版 |
|-----------|------------|
| 16 周学期 | 10-15 天集中营 |
| 研究生独立完成 | 分步引导 + 模板代码 |
| 工具精通（Altium 全流程） | 工具入门（知道这个工具做什么、怎么用） |
| 完整 PCB 设计制造 | 理解设计流程 + 使用开发板/模块 |
| FreeRTOS 多线程 | 理解并发概念 + 简单任务调度 |
| 自主云端集成 | 跟随教程完成基本云连接 |
| 口试答辩 30min | 5-10 min 展示 + 问答 |

### 6.2 作业量与难度映射

每个项目的作业量应根据**难度等级**调整：

| 难度 | 代表项目 | 作业数量 | 作业深度 |
|------|---------|---------|---------|
| Easy | weather-cube | 3-4 份 | 侧重 Layer 1+2 |
| Mid | env-monitor, desktop-pet | 4-6 份 | 侧重 Layer 1+2，部分 Layer 3 |
| High | oscilloscope, robot, satellite | 5-7 份 | Layer 1+2+3 均涉及 |
| XHigh | electronbot | 6-8 份 | 全面覆盖，Layer 3 要求更高 |

### 6.3 "对话式理解"考核标准

高中生项目考核的核心不是"做得多好"，而是"能不能说清楚为什么"：

- ✅ 能解释电源线路为什么这样设计（比如为什么用 LDO 而不是 Buck）
- ✅ 能解释传感器选型的考量（精度 vs 成本 vs 功耗的权衡）
- ✅ 能解释通信协议的选择理由（为什么用 I2C 而不是 SPI）
- ✅ 能解释代码架构的设计思路（为什么分模块、为什么用中断而不是轮询）
- ❌ 不要求独立从零设计完整电路
- ❌ 不要求精通所有底层寄存器操作
- ❌ 不要求完整实现所有高级功能

## 七、作业设计的核心模板 | Core Assignment Design Template

基于以上抽象，每个项目的每份作业应遵循以下模板：

```markdown
# [项目名] - 作业 [编号]: [标题]

## 学习成果 | Learning Outcomes
- 列出 2-4 条具体的、可衡量的学习成果
- 使用 Bloom 动词：识别、解释、实现、分析、设计

## 为什么学这个 | Why This Matters
- 用通俗语言解释这个知识点在整个项目中的位置
- 类比生活经验帮助理解

## 前置知识 | Prerequisites
- 列出需要的先备知识和学习资源链接

## 任务清单 | Task List
### 任务 1: [名称] (难度: ⭐)
- 具体操作步骤
- 预期产出
- 验证方法

### 任务 2: [名称] (难度: ⭐⭐)
- 具体操作步骤
- 预期产出
- 验证方法

### ...

## 理解验证问题 | Comprehension Check
- 3-5 个"为什么"类问题
- 提供参考答案
- 侧重设计选择的权衡与理由

## 提交要求 | Submission Requirements
- 具体的文件/截图/代码要求
- 代码规范检查清单

## 评分标准 | Rubric
| 维度 | 满分 | 评分细则 |
|------|------|---------|
```

## 八、作业序列设计原则 | Assignment Sequencing Principles

1. **每个作业的输出 = 下一个作业的输入**（Pipeline 原则）
2. **先理解后实践**：每个新概念先有阅读/观察任务，再有动手任务
3. **个人任务先于团队任务**：工具技能先单独练好
4. **每个阶段末尾有检查点**：Weekly Check-in 或阶段性 Review
5. **Errata 机制**：每个作业文档预留常见问题和解决方案的更新空间
6. **难度曲线平滑**：相邻作业之间的难度跳跃不超过一个 Bloom 层级
