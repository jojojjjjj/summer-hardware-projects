# 作业设计标准与评审规范 | Assignment Design Standards & Review Rubric

> 本文档基于 UPenn ESE5160 课程设计思路抽象，作为所有项目作业设计的**统一标准**和评审团的**审核依据**。

---

## 一、核心设计原则 | Core Design Principles

### 1.1 三层知识深度模型（必须遵循）

每份作业必须明确覆盖以下三层，但深度根据项目难度调整：

| 层级 | 含义 | 高中生要求 | 考核形式 |
|------|------|-----------|---------|
| **Layer 1: Know-What** | 知道是什么 | 每个方向**必须覆盖** | 识记、描述、画图 |
| **Layer 2: Know-Why** | 知道为什么 | **重点培养**——能和老师进行技术对话的关键 | "理解验证问题"（Comprehension Check）|
| **Layer 3: Know-How** | 知道怎么做 | 适度——能完成基本实现 | 实际操作、代码提交 |

**核心标准**：学生能回答"为什么这样设计"比"做得有多完美"更重要。

### 1.2 作业管道原则（Pipeline Principle）

- 每份作业的**输出 = 下一份作业的输入**
- 作业之间形成严格的依赖链，不是孤立的任务
- 例：HW1 选器件 → HW2 验证器件 → HW3 画原理图 → HW4 PCB布局

### 1.3 先理解后实践原则

- 每个新概念**先有阅读/观察任务**，再有动手任务
- 每份作业的"理解验证问题"必须在动手任务之前完成

## 二、作业数量标准 | Assignment Quantity Standards

根据项目难度，作业数量遵循以下规则：

| 难度 | 项目 | 正式作业数 | 每日课后任务 | Check-in报告 |
|------|------|-----------|------------|-------------|
| **Easy** | project-06-weather-cube | 3 份 | 每天1个简短任务 | 1份Week 1 |
| **Mid** | project-02-env-monitor, project-05-desktop-pet | 4-5 份 | 每天1个任务 | 2份 (Week 1 + Week 2) |
| **High** | project-01, project-04, project-07, project-09 | 5-7 份 | 每天1-2个任务 | 2份 + Phase Review |
| **XHigh** | project-08-electronbot | 6-8 份 | 每天1-2个任务 | 2份 + Phase Review |
| **20天** | project-03-4mode-keyboard | 8-10 份 | 每天1个任务 | 3份 + Phase Review |

### 作业深度分布

| 难度 | Layer 1 占比 | Layer 2 占比 | Layer 3 占比 |
|------|-------------|-------------|-------------|
| Easy | 50% | 40% | 10% |
| Mid | 40% | 35% | 25% |
| High | 30% | 35% | 35% |
| XHigh | 25% | 35% | 40% |

## 三、作业标准化模板 | Standardized Assignment Template

每份作业**必须**遵循以下模板（参考宾大 ESE5160 结构）：

```markdown
# [项目名] - 作业 [编号]: [标题]

## 学习成果 | Learning Outcomes
- 列出 2-4 条具体的、可衡量的学习成果
- 使用 Bloom 动词：识别(Identify)、解释(Explain)、实现(Implement)、分析(Analyze)、设计(Design)

## 为什么学这个 | Why This Matters
- 用通俗语言解释这个知识点在整个项目中的位置
- 类比生活经验帮助理解（至少1个日常类比）

## 前置知识 | Prerequisites
- 列出需要的先备知识和学习资源链接

## 任务清单 | Task List

### 任务 1: [名称] (难度: ⭐)
- 具体操作步骤（细到"点击哪里""输入什么"）
- 预期产出（具体的文件/截图/代码）
- 验证方法（如何确认做对了）

### 任务 2: [名称] (难度: ⭐⭐)
- 具体操作步骤
- 预期产出
- 验证方法

### ...

## 理解验证问题 | Comprehension Check
- 3-5 个"为什么"类问题（**这是最核心的部分**）
- 每个问题提供参考答案
- 侧重设计选择的权衡与理由
- 例：为什么这个电源线路用LDO而不是Buck？为什么选I2C而不是SPI？

## 提交要求 | Submission Requirements
- 具体的文件/截图/代码要求
- 代码规范检查清单

## 评分标准 | Rubric
| 维度 | 满分 | 评分细则 |
|------|------|---------|
```

## 四、理解验证问题设计规范 | Comprehension Check Design Specification

这是区别于普通课程设计的**核心特色**，也是评审团审核的重点。

### 4.1 问题类型分布

每份作业的理解验证问题应包含以下类型：

| 类型 | 占比 | 示例 |
|------|------|------|
| **设计权衡题** | 40% | "为什么用LDO而不是Buck做3.3V稳压？" |
| **原理追问题** | 30% | "I2C的上拉电阻为什么不能太小也不能太大？" |
| **故障分析题** | 20% | "如果传感器读数全是0，可能的原因有哪些？按可能性排序" |
| **类比迁移题** | 10% | "把SPI通信类比成寄信，CS/CLK/MOSI/MISO分别对应什么？" |

### 4.2 问题质量标准

- ❌ 差：什么是I2C？（纯识记，无思考）
- ⚠️ 中：I2C和SPI有什么区别？（有对比，但缺乏深层理解）
- ✅ 好：这个项目为什么选择I2C连接温湿度传感器而不是SPI？从引脚数量、速度需求、多设备场景三个角度分析。（具体场景+多维度分析）

### 4.3 每份作业的最低问题数量

| 难度 | 理解验证问题数 |
|------|--------------|
| Easy | 2-3 个 |
| Mid | 3-4 个 |
| High/XHigh | 4-5 个 |

## 五、评分标准设计规范 | Rubric Design Specification

### 5.1 总体评分维度（参考宾大 ESE5160 + AGENTS.md）

| 维度 | 权重 | 说明 |
|------|------|------|
| 技术实现 | 35-40% | 功能完整度、代码质量、稳定性 |
| 文档质量 | 15-20% | README、注释、协议文档 |
| 演示展示 | 15-20% | Final Demo 清晰度与深度 |
| 进度汇报 | 10-15% | 每周 Check-in 质量 |
| 团队协作 | 5-10% | Git 提交记录、分工合理性 |

### 5.2 单份作业评分维度

每份作业的评分表必须包含：

| 维度 | 必须包含 |
|------|---------|
| **完成度** | 任务是否全部完成 |
| **理解深度** | 理解验证问题的回答质量（**权重最高**）|
| **代码/文档质量** | 代码可编译、注释清晰、文档完整 |
| **创意拓展** | （可选）超出要求的额外探索 |

### 5.3 评分粒度

- 每个评分项给出**明确的分数和标准描述**
- 区分：满分（完全达到）、部分分（有遗漏）、零分（未完成）
- 总分建议使用100分制，便于横向比较

## 六、评审团审核标准 | Review Panel Audit Criteria

评审团对每份作业设计进行审核时，使用以下检查清单：

### 6.1 结构完整性（30分）

- [ ] 是否包含 Learning Outcomes（2-4条，使用Bloom动词）
- [ ] 是否包含 Why This Matters（有日常类比）
- [ ] 是否包含 Prerequisites（前置知识清单）
- [ ] 任务清单是否分难度标注（⭐/⭐⭐/⭐⭐⭐）
- [ ] 每个任务是否有：操作步骤 + 预期产出 + 验证方法
- [ ] 是否包含 Comprehension Check（理解验证问题）
- [ ] 是否包含 Submission Requirements
- [ ] 是否包含 Rubric 评分标准

### 6.2 知识深度合理性（30分）

- [ ] Layer 1（Know-What）是否覆盖该作业涉及的所有基础概念
- [ ] Layer 2（Know-Why）是否有足够的设计权衡题和原理追问题
- [ ] Layer 3（Know-How）的难度是否适合高中生（不超过大二水平）
- [ ] 三层深度比例是否符合项目难度要求

### 6.3 作业链连续性（20分）

- [ ] 本作业的输出是否是下一作业的输入
- [ ] 作业之间的难度跳跃是否不超过一个Bloom层级
- [ ] 阶段性检查点（Check-in）是否覆盖该阶段的关键能力

### 6.4 高中生适配度（20分）

- [ ] 步骤描述是否足够细致（零基础可跟随）
- [ ] 是否有常见错误预警（Errata/常见坑）
- [ ] 预计完成时间是否合理（课后1-2小时）
- [ ] 理解验证问题是否用通俗语言描述
- [ ] 是否提供了参考答案

## 七、特殊作业类型规范 | Special Assignment Types

### 7.1 进度汇报（Check-in Report）

```markdown
# [项目名] - 第 [N] 周进度汇报

## 汇报要求
- 字数：600-1200字（中文或英文）
- 必须包含：
  1. 已完成的功能列表（配截图/照片）
  2. 遇到的主要困难和解决方案（至少3个技术问题，格式：现象→原因分析→解决方法）
  3. 下阶段计划（逐日列出预期目标）
  4. 个人反思（学到了什么、哪里可以做得更好）

## 评分重点
- 内容的详细程度和真实性（40%）
- 问题分析的深度（30%）
- 计划的可行性（20%）
- 反思的真诚度（10%）
```

### 7.2 最终演示（Final Presentation）

```markdown
# [项目名] - 最终演示

## 演示要求
- 时长：5-8分钟口头演示 + 2-3分钟Q&A
- 必须包含：
  1. 项目背景和目标（1分钟）
  2. 技术架构概述（1分钟）
  3. 硬件实物展示 + 功能演示（2-3分钟）
  4. 关键技术点讲解（1-2分钟）——重点解释"为什么这样设计"
  5. 遇到的问题与解决方案（1分钟）

## 评分标准
- 演示清晰度（30%）
- 功能完整度（25%）
- Q&A 应对能力——特别是"为什么"类问题（25%）
- 时间控制（10%）
- PPT/材料质量（10%）
```

### 7.3 项目文档 + Demo网站

```markdown
# [项目名] - 项目文档 + Demo网站

## 文档要求
- README.md 完整、清晰
- 代码有适当注释
- 接线图/原理图清晰
- 包含调试指南

## Demo网站要求
- 项目介绍页
- 功能展示（图片/视频/GIF）
- 技术架构说明
- 制作过程和心得
- 部署到 GitHub Pages / Vercel / Cloudflare Pages
```

## 八、作业序列设计示例（参考宾大 ESE5160）

### 8.1 宾大 ESE5160 的渐进式序列

```
A00G(概念+框图) → A01G(器件+电源) → A02G(原理图I) → A03G(原理图II)
→ A04G(PCB布局) → A05G(制造发布) → A06G(FreeRTOS+I2C) → A07G(板卡调试)
→ A08G(云端) → A09G(驱动+应用) → A10G(审判日) → A11G(最终提交) → A12G(口试)
```

**关键洞察**：
1. 前5份作业（A00-A05）是**硬件设计**，形成完整的硬件Pipeline
2. 后5份作业（A06-A10）是**固件+云**，形成完整的软件Pipeline
3. 最后2份（A11-A12）是**交付+答辩**
4. 个人作业（P系列）与团队作业（G系列）双轨并行

### 8.2 高中生项目的典型序列

```
HW1(环境搭建+概念理解) → HW2(硬件基础/器件认识) → HW3(核心模块驱动)
→ HW4(数据/通信/控制) → HW5(系统集成) → HW6(最终展示+文档)
→ Check-in Reports(阶段检查) → Final Presentation(答辩)
```

**差异点**：
- 不需要完整PCB设计制造，用开发板+模块
- 不需要深入寄存器级别，理解API层即可
- 侧重点从"做出来"转向"说清楚为什么"
- 增加理解验证问题的比重
