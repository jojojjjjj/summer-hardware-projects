## 理解验证问题 | Comprehension Check

**Q1（设计权衡题）**：ElectronBot 选择 STM32F4 而不是 ESP32。从实时性、USB 外设、功耗三个角度分析原因。

> **参考答案**：1）实时性：STM32F4 的 Cortex-M4 有 NVIC 中断控制器和确定性执行时间，适合精确的舵机 PID 控制循环（200Hz+）；ESP32 的 FreeRTOS 调度有更大抖动。2）USB 外设：STM32F4 内置全速/高速 USB OTG PHY，可直接实现 USB CDC 设备；ESP32 的 USB 需要外部芯片或使用较新的 ESP32-S3。3）功耗：STM32F4 在全速运行时约 90mA@168MHz，ESP32 约 200mA@240MHz，ElectronBot 有 USB 供电所以功耗不是主要考量，实时性和 USB 是决定因素。

**Q2（原理追问题）**：原理图中 USB D+ 线上有一个 1.5kΩ 上拉电阻接到 VDD。这个电阻的作用是什么？如果没有它，PC 能识别这个 USB 设备吗？

> **参考答案**：USB 协议通过 D+ 上的上拉电阻来标识设备速度：1.5kΩ 上拉到 3.3V 表示全速设备（12Mbps），D- 上拉表示低速设备（1.5Mbps）。没有这个上拉电阻，PC 无法检测到设备连接（USB 主机通过检测 D+/D- 被拉高来判断设备插入），设备不会被枚举。这个电阻本质上是 USB 的"门铃"——设备拉响门铃告诉主机"我来了"。

**Q3（故障分析题）**：STM32CubeIDE 编译成功但烧录后程序不运行（LED 不亮）。列出至少 3 种可能原因。

> **参考答案**：1）时钟配置错误（HSE/HSI 未正确配置导致系统时钟异常）；2）ST-Link 连接模式不对（应选 SWD 而非 JTAG）；3）BOOT0 引脚为高电平（进入 bootloader 而非用户程序）；4）GPIO 配置与实际 LED 引脚不匹配。

**Q4（类比迁移题）**：把 STM32 的中断系统类比成餐厅服务。NVIC、中断优先级、中断处理函数分别对应什么？

> **参考答案**：NVIC = 餐厅经理（决定哪些请求该优先处理）；中断优先级 = 客人的重要程度（VIP 优先）；中断处理函数 = 服务员的应对流程（接到请求后具体做什么）。高优先级中断可以打断低优先级 = VIP 可以插队。

# HW2: PCB 焊接质量考核与硬件集成 | PCB Soldering Assessment & Hardware Integration

> **截止日期**：Day 4 | **权重**：12% | **预计耗时**：课内完成为主

## 学习成果 | Learning Outcomes

1. **解释** QFN 封装焊接的技术要点和常见缺陷
2. **实现** ElectronBot 主板和舵机驱动板的完整焊接
3. **完成** 首次上电测试和舵机标定
4. **分析** 焊接缺陷的成因和返修策略

## 为什么学这个 | Why This Matters

焊接是硬件项目从"图纸"到"实物"的关键一步。QFN 封装（无引脚底部焊盘）是工业界最常见的芯片封装之一，掌握它的焊接技巧非常有价值。焊接质量直接决定硬件是否能正常工作。

**日常类比**：焊接就像烹饪——食材（元器件）一样，但火候（温度）和时间掌握不同，结果可能天差地别。QFN 焊接就像煎蛋——表面看着简单，但翻面时机不对就碎了。

- 完成舵机驱动板焊接
- **预期产出**：焊接成品照片（正面+反面）

### 任务 2: 首次上电测试 (难度: ⭐⭐)
- 用万用表检查关键电压点：3.3V、5V、VDD
- 连接 ST-Link 烧录测试程序
- **预期产出**：电压测量记录 + 烧录成功截图

### 任务 3: 舵机标定 (难度: ⭐⭐)
- 通过 I2C 发送指令控制单个舵机旋转
- 记录舵机的角度范围和响应延迟
- **预期产出**：舵机标定数据 + 动作视频

## 理解验证问题 | Comprehension Check

**Q1（设计权衡题）**：QFN 封装比 THT（通孔）封装难焊得多。为什么 ElectronBot 还要使用 QFN 封装的 STM32？从尺寸、散热、高频性能三个角度分析。

> **参考答案**：1）尺寸：QFN 封装面积仅为同引脚数 THT 的 1/5-1/10，ElectronBot 主板空间有限必须用小封装。2）散热：QFN 的底部散热焊盘直接贴 PCB，热阻远低于 THT，168MHz 运行时散热更可靠。3）高频性能：QFN 引脚极短（<1mm），寄生电感小，适合 USB 和高速 SPI 信号；THT 引脚长，寄生参数大，高速信号完整性差。代价是焊接难度和返修难度增大。

**Q2（原理追问题）**：焊接后用万用表测量 3.3V 电源和 GND 之间电阻只有几欧姆。这说明什么问题？可能的原因有哪些？

> **参考答案**：说明电源和地之间短路（正常应>1kΩ）。可能原因：1）QFN 芯片底部焊锡桥连了相邻的电源和地焊盘；2）0402/0201 电容焊歪导致两端短路；3）PCB 制造缺陷（内层短路）。排查方法：逐步割断电源走线缩小短路范围，或用热成像仪找发热点。

**Q3（故障分析题）**：某个舵机抖动严重，无法稳定停在目标位置。列出至少 3 种可能原因。

> **参考答案**：1）PID 参数不当（Kp 过大导致震荡）；2）I2C 通信不稳定（命令丢失或延迟）；3）舵机齿轮间隙（机械游隙导致位置无法精确保持）；4）电源电压不稳（舵机动作时拉低供电导致控制器复位）。

## 理解验证问题 | Comprehension Check

**Q1（设计权衡题）**：ElectronBot 用 USB CDC 而不是传统 UART 串口与 PC 通信。USB CDC 相比 UART 有哪些优势？在什么场景下 UART 更合适？

> **参考答案**：USB CDC 优势：1）速度：全速 USB 12Mbps 远超 UART 115200bps；2）可靠性：USB 有 CRC 校验和重传机制，UART 无硬件级错误检测；3）即插即用：不需要设置波特率等参数。UART 更合适的场景：1）MCU 之间短距离通信（不需要 USB 协议栈）；2）调试阶段（UART 更简单，USB 协议栈出问题时无法调试）；3）超低功耗场景（USB 枚举功耗远高于 UART）。

**Q2（原理追问题）**：GC9A01 是圆形屏幕（240x240），但 SPI 传输是线性的（逐行逐像素）。如何实现圆形显示区域之外的部分"透明"（不显示）？

> **参考答案**：方法一：在帧缓冲中，对于超出圆形区域的像素（距中心 > 120像素），写入黑色（0x0000），GC9A01 的黑色就是"不发光"的效果。方法二：利用 GC9A01 的局部更新功能（设置显示窗口），只更新圆形内的像素区域，跳过四角。方法一更简单但四角会显示为黑色区域；方法二效率更高但代码更复杂。ElectronBot 使用方法一因为圆形 LCD 的四角本来就没有像素（物理上是圆形）。

**Q3（故障分析题）**：USB 设备插入 PC 后显示"未知 USB 设备（设备描述符请求失败）"。可能的原因有哪些？

> **参考答案**：1）USB 描述符配置错误（VID/PID/端点描述符有误）；2）D+/D- 线接反或上拉电阻缺失；3）USB 时钟未正确配置（STM32F4 USB 需要 48MHz 精确时钟）；4）VBUS 检测引脚未连接，设备不知道已插入主机。

**Q4（设计权衡题）**：屏幕渲染用帧缓冲（先画好再整体发送）还是直接逐像素写（边画边显示）？各有什么优缺点？

> **参考答案**：帧缓冲优点：1）无闪烁（整体刷新）；2）可以实现双缓冲（前台显示+后台绘制）；3）便于复杂图形合成（多层叠加后一次性输出）。缺点：需要 240×240×2=115KB RAM，STM32F4 的 192KB SRAM 占用超 60%。直接写优点：内存占用极小（不需要帧缓冲）；缺点：1）画面撕裂（上半部分是旧帧、下半部分是新帧）；2）复杂图形需要多次 SPI 传输，效率低。ElectronBot 选择帧缓冲+部分刷新的折中方案。

**Q5（类比迁移题）**：把 USB 枚举过程类比成入职手续。设备插入、描述符返回、驱动加载、设备就绪分别对应入职的什么步骤？

> **参考答案**：设备插入 = 新员工到前台报到（主机检测到有设备）；描述符返回 = 填写入职登记表（告诉主机我是谁、能做什么、需要什么资源）；驱动加载 = 分配工位和电脑（操作系统准备软件接口）；设备就绪 = 开始上班（可以正常通信）。

# HW4: I2C 舵机 PID 闭环与表情动画系统 | Servo PID Control & Expression Animation

> **截止日期**：Day 7 | **权重**：15%

## 学习成果 | Learning Outcomes

**Q1（设计权衡题）**：ElectronBot 使用带 PID 闭环的智能总线舵机，而不是普通的 PWM 舵机。PID 闭环比开环控制好在哪里？代价是什么？

> **参考答案**：PID 闭环优点：1）位置精确——即使有外力推动，舵机能自动回到目标位置；2）速度可控——可以平滑加减速而非瞬间到位；3）力矩可控——可以控制"用多大力"而不是"转多快"。代价：1）需要编码器反馈，硬件成本更高；2）PID 参数需要调校，增加开发工作量；3）通信协议更复杂（I2C 命令帧 vs 简单 PWM 信号）。对于桌面宠物机器人，精确的位置控制和力控是做出自然动作的关键，代价是值得的。

**Q2（原理追问题）**：6 自由度机器人的正运动学计算是什么意思？如果只需要让机器人挥手，为什么也需要运动学计算？

> **参考答案**：正运动学是"已知各关节角度，求末端执行器（手掌）的空间位置"。即使简单的挥手动作也需要协调肩、肘、腕多个关节的角度——如果肩关节转 30° 而肘关节不动，手的轨迹和两个关节协调转动的轨迹完全不同。运动学计算帮助理解"每个关节该转多少度才能让手到达期望位置"。不计算运动学的话，只能靠试错调角度，效率极低且无法复现。

**Q3（故障分析题）**：舵机发出"嗡嗡"声但不动。与"抖动"（小幅震荡）有什么区别？可能的原因分别是什么？

> **参考答案**："嗡嗡"但不动 = 舵机在用力但被卡住或力矩不足，常见原因：1）目标位置超出物理限位；2）负载过重（手臂被卡住）；3）供电电压不足导致力矩不够。"抖动" = 舵机在目标位置附近来回震荡，常见原因：1）Kp 过大导致超调震荡；2）I2C 通信延迟导致反馈滞后；3）编码器噪声导致目标角度不稳定。两者的本质区别：嗡嗡=力矩不够/受阻，抖动=控制不稳定。

**Q4（设计权衡题）**：表情动画用帧动画（预制作图片序列逐帧播放）还是实时渲染（用代码实时绘制图形）？各适合什么场景？

> **参考答案**：帧动画优点：1）效果精美（可以用专业工具制作）；2）播放只需按序显示，CPU 占用低；3）帧数可控，时序精确。缺点：占用 Flash 空间大（每帧 240×240×2=115KB，10 帧=1.15MB）。实时渲染优点：1）几乎不占 Flash（只需代码）；2）参数可动态调整（瞳孔位置随传感器变化）；3）可做无限平滑的过渡。缺点：CPU 占用高，复杂图形计算量大。ElectronBot 的策略是混合使用：背景用帧动画（精美的表情素材），瞳孔位置用实时渲染（跟随传感器动态变化）。

> **参考答案**：1）USB 库兼容性：底层 USB 库（libusb）是 C 语言编写的，C# 无法直接调用；DLL 提供了 C++→C# 的接口转换。2）实时性：C++ 层可以独立于 Unity 的帧率运行高频率的 USB 通信，不受 Unity 垃圾回收和帧率波动影响。3）跨平台：DLL 层封装了平台相关的 USB 调用，上层 C# 代码无需关心底层差异。4）解耦：DLL 层和 Unity 层可以独立开发、独立测试、独立更新。

**Q2（原理追问题）**：OpenCV 人脸检测使用 Haar 级联分类器。简单解释它的工作原理——为什么它能从像素中识别出人脸？

> **参考答案**：Haar 级联的原理是"特征匹配+级联淘汰"。它利用人脸的结构特征：眼睛区域比脸颊暗（明暗对比特征），鼻梁比两侧亮等。这些特征用 Haar 小波（黑白相邻矩形）来提取。级联的意思是多级过滤：第一级快速排除明显不是人脸的区域（如纯色区域），后续级逐步细化判断。只有通过所有级的区域才被判定为人脸。就像多轮面试——第一轮简历筛选（快速排除），第二轮笔试（初步评估），第三轮面试（精细判断）。

> **参考答案**：1）DLL 版本不匹配——Unity 引用的 DLL 版本与固件的通信协议不一致（命令格式变了）；2）USB 端口独占——串口助手未关闭导致 DLL 无法打开同一端口；3）数据格式错误——Unity 发送的数据格式（如浮点数字节序）与固件期望的格式不匹配；4）DLL 初始化未完成——Unity 在 DLL 完成 USB 枚举前就发送了数据。

**Q4（设计权衡题）**：OpenCV 处理一帧图像大约需要 30-50ms，但舵机控制需要 5ms 级别的响应。如何解决这个时序不匹配？

> **参考答案**：使用异步架构：视觉处理在独立线程/进程中运行（10-30fps），将检测结果（如人脸位置坐标）放入共享缓冲区；舵机控制循环在另一个线程中以 200Hz 运行，从缓冲区读取最新的目标位置并平滑插值。这样视觉"看一眼"的间隔约 33ms，但舵机"动一下"的间隔仅 5ms，中间的过渡由插值算法填充。这是嵌入式系统中常见的"慢感知+快控制"架构。

**Q1（设计权衡题）**：LLM API 调用放在 PC 端而不是 STM32 端。为什么？如果要在 STM32 上直接调用 LLM，需要什么条件？

> **参考答案**：STM32F4 的 192KB SRAM 和 1MB Flash 远远不够运行任何 LLM（最小模型也需数百 MB 内存）。API 调用需要 HTTPS 通信，STM32F4 的资源也难以支撑 TLS 握手。在 STM32 上直接调用 LLM 需要：1）至少 256MB RAM 的 MCU/SoC（如树莓派级别）；2）网络模块支持 HTTPS；3）足够的 Flash 存储模型权重或稳定的互联网连接。ElectronBot 的设计是"嵌入式采集+云端推理+桌面呈现"的典型边缘计算架构。

**Q2（原理追问题）**：System Prompt 对 LLM 的回答质量有很大影响。一个好的 System Prompt 应该包含哪些要素？写一个 ElectronBot 的 System Prompt 示例。

> **参考答案**：好 System Prompt 的要素：1）角色定义（你是一个桌面宠物机器人）；2）性格特征（活泼、好奇、有点调皮）；3）回答风格（简短、口语化、不用专业术语）；4）限制条件（回答不超过 50 字、不能说脏话）；5）上下文信息（你叫 ElectronBot、你有一个圆形屏幕可以显示表情）。示例："你是 ElectronBot，一个桌面宠物机器人。你活泼、好奇、有点调皮。你用简短的口语化中文回答，每次不超过 50 字。你有圆形屏幕可以显示表情。用户通过语音和你对话。"

**Q3（故障分析题）**：语音交互延迟很高（从说话到机器人回答超过 5 秒）。分析延迟来自哪些环节，如何优化？

> **参考答案**：延迟分解：STT 约 1-2 秒 + LLM 首 token 约 1-3 秒 + TTS 约 0.5-1 秒 + USB 传输约 0.1 秒 = 总计 2.6-6.1 秒。优化方案：1）使用流式 LLM API（收到首 token 就开始 TTS，不等完整回答）；2）选择更快的 STT 模型（如 Whisper tiny）；3）本地运行小模型（如 1B 参数模型）替代云端 API；4）优化 System Prompt 减少 LLM 生成长度。

**Q4（设计权衡题）**：回顾整个项目，ElectronBot 横跨"固件+桌面软件+AI"三个领域。这种跨领域架构的优势和挑战分别是什么？

> **参考答案**：优势：1）能力远超单领域项目（固件控制硬件+Unity 做界面+AI 做交互）；2）学生接触真实的产品架构（现代产品都是多系统协作）；3）每个人都能找到感兴趣的方向。挑战：1）调试难度指数级增长（固件 bug 可能是 USB 协议问题，也可能是 Unity 逻辑错误）；2）接口定义和版本管理复杂（固件协议改了，Unity 也要改）；3）学习曲线陡峭（STM32 HAL/C/Unity C#/Python OpenCV 三个技术栈）。
